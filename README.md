# Project 4: SQL Deep Dive — Brazilian E-Commerce

A project demonstrating production-quality PostgreSQL skills across four tiers of query complexity, query optimization patterns, and an index strategy — all on a realistic mock of the Olist Brazilian E-Commerce dataset.

---

## Dataset

Mock data generated by `scripts/load_data.py`.  All distributions mirror the real Olist dataset proportions.

| Table           | Rows    | Description                                    |
|-----------------|---------|------------------------------------------------|
| customers       | 10,000  | Customer location (city, state, zip)           |
| sellers         | 500     | Seller location                                |
| products        | 3,000   | Category, dimensions, price                    |
| orders          | 100,000 | Status, timestamps, customer FK                |
| order_items     | ~120,000| Per-item price + freight, seller + product FKs |
| order_reviews   | ~95,000 | Score 1-5, optional comment                    |
| order_payments  | ~110,000| Type, installments, value                      |

Date range: 2016-09-01 to 2018-08-31

Key distributions used in generation:
- Order status: delivered 78%, shipped 8%, invoiced 5%, canceled 4%, processing 3%, other 2%
- Payment type: credit_card 74%, boleto 19%, voucher 5%, debit_card 2%
- Review score: 5-star 59%, 4-star 19%, 3-star 8%, 1-star 11%, 2-star 3%

## Quick Start

```bash
# 1. Start PostgreSQL (port 5433 to avoid conflicts)
docker-compose up -d

# 2. Install Python dependencies
pip install -r requirements.txt

# 3. Generate mock data and load into the database
python scripts/load_data.py

# 4. Run all queries and capture output
python scripts/run_queries.py

# 5. Open the notebook
jupyter notebook notebooks/sql_deep_dive.ipynb
```

## Query Cookbook

### Tier 1 — Basic (sql/tier1_basic.sql)

| # | Business Question | Techniques |
|---|-------------------|------------|
| 1 | Total revenue and order count | SUM, COUNT, WHERE |
| 2 | Average order value by state | GROUP BY, ROUND |
| 3 | Top 10 categories by revenue | JOIN, ORDER BY, LIMIT |
| 4 | Orders per month | DATE_TRUNC, time series |
| 5 | Customers above 90th percentile spend | PERCENTILE_CONT, HAVING subquery |

### Tier 2 — Intermediate (sql/tier2_intermediate.sql)

| # | Business Question | Techniques |
|---|-------------------|------------|
| 1 | Revenue by state × category | 4-table JOIN, GROUP BY two dimensions |
| 2 | Monthly revenue + YoY growth | CTE, self-join on LAG year, NULL-safe division |
| 3 | Customers in both 2017 and 2018 | Double IN subquery, EXTRACT |
| 4 | Daily order count + 7-day rolling average | CTE, AVG OVER ROWS BETWEEN |
| 5 | Cohort retention by first-order month | Two-CTE pipeline, DATE_TRUNC, AGE |

### Tier 3 — Advanced (sql/tier3_advanced.sql)

| # | Business Question | Window Function |
|---|-------------------|-----------------|
| 1 | Top 3 customers per state by LTV | ROW_NUMBER OVER PARTITION BY state |
| 2 | 2nd highest-revenue product per category | RANK + DENSE_RANK |
| 3 | Day-over-day revenue change % | LAG / LEAD |
| 4 | Running total revenue per customer | SUM OVER ROWS BETWEEN UNBOUNDED PRECEDING |
| 5 | Customer RFM segmentation | NTILE(5) across recency, frequency, monetary |
| 6 | Star schema: state × category × year | 4-table join, multi-dimension GROUP BY |

### Tier 4 — Optimization (sql/tier4_optimization.sql)

| # | Problem Pattern | Fix |
|---|-----------------|-----|
| 1 | Sequential scan on orders by customer + date | Composite index (customer_id, purchase_timestamp DESC) |
| 2 | Correlated subquery for per-category avg price | Window function AVG OVER PARTITION BY |
| 3 | N+1 subqueries for seller stats | Single aggregating LEFT JOIN |
| 4 | Missing composite index for state×category×month | CTE pre-filter + covering indexes |

## Index Strategy (sql/index_strategy.sql)

| Index | Columns | Justification |
|-------|---------|---------------|
| idx_orders_customer_date | (customer_id, purchase_timestamp DESC) | Customer history and LTV queries |
| idx_orders_status | (status) WHERE status != 'delivered' | Operational dashboards on non-delivered orders |
| idx_order_items_product | (product_id) | Category revenue joins |
| idx_order_items_seller | (seller_id) | Seller performance joins |
| idx_order_items_order | (order_id) | Payment and review joins back to items |
| idx_payments_order | (order_id) | Payment-type breakdowns |
| idx_reviews_order | (order_id) | Review score aggregations |
| idx_products_category | (category) | Category filter and GROUP BY |
| idx_customers_state | (state) | State segmentation |
| idx_orders_purchase_ts | (purchase_timestamp DESC) | Time-series and monthly rollups |

---

## Key Learnings

1. Composite indexes eliminate sequential scans on multi-column filter + sort patterns.
2. Window functions (AVG/SUM OVER PARTITION BY) replace O(n) correlated subqueries with a single pass.
3. Pre-filtering with a CTE reduces the join input size before multi-table aggregations.
4. Partial indexes on high-cardinality filtered columns (e.g., status != 'delivered') stay small and cache-resident.
5. NTILE-based RFM scoring in a CTE pipeline produces customer segments in a single query.
6. EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) is the primary tool for confirming that planner estimates match actual row counts.
